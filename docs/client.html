TCP Client — Behaviour & Monitoring Reference

1. What is tcp.client.monitoring.tick-ms?

This is the interval (in milliseconds) at which the client's background health-check scheduler fires. Controlled by the @Scheduled annotation on TcpClientService.monitorConnection(): 

@Scheduled(fixedDelayString  = "${tcp.client.monitoring.tick-ms:2000}",
           initialDelayString = "${tcp.client.monitoring.tick-ms:2000}")
public void monitorConnection() { ... }

fixedDelay (not fixedRate) means the next tick starts after the previous one finishes, so the real gap is tick-ms + time-to-run-probe.

2. What happens every tick?

Tick flow
  1. Guard: if monitoringEnabled = false → skip immediately (no cost).
  2. Guard: if state is not CONNECTED (e.g. CONNECTING, ERROR) → skip immediately.
  3. Acquire the object-level synchronized lock on TcpClientService.
  4. Send probe:shallow over the live socket to the server.
  5. Wait up to probe-timeout-ms (default 1000 ms) for a response.
  6. If the response contains "OK" → log success, release lock.
    If timeout or no OK → call handleConnectionError(), which:
    • Transitions state to ERROR
    • Closes the socket
    • Spawns the background reconnection-thread

3. How the client behaves overall

PhaseWhat the client does
Startup Reads config from env vars / YAML. In terminal mode calls connectToServer() immediately. In ui mode waits for user action.
Connecting Tries socket.connect(host, port, connectTimeout). On failure waits waitBetweenRetries (170 ms) and retries — fixed wait, no exponential back-off. Max retries = 99 999 (effectively forever).
Connected Holds one persistent socket. Keeps a SynchronousQueue for request/response pairing. Spring Integration's event listener detects drops and triggers immediate reconnection. Background health tick also validates liveness.
Drop detected (event) TcpConnectionEventListener fires synchronously on the socket reader thread. Transitions state → ERROR → DISCONNECTED, then triggers reconnection-thread within milliseconds.
Drop detected (probe) monitorConnection() sends a shallow probe; if no OK within probe-timeout-ms it closes the socket and triggers reconnection. This is the fallback path — the event listener normally beats it.
Reconnecting Fixed 170 ms loop. Each attempt creates a fresh TCP socket with a 150 ms connect timeout (enforced at the socket level, not Spring level). As soon as the server accepts, state → CONNECTED.

4. Current setting: tick-ms = 2000

Behaviour at 2000 ms
  • One probe every ≈2 s while connected.
  • Each probe sends one small TCP message (probe:shallow) and waits up to 1 s for reply.
  • Worst-case extra probe cost per 2 s window: ~1 s (probe timeout). Typical: a few ms RTT.
  • Scheduler occupies one thread from the SchedulingConfig pool (size 2) for <10 ms normally.
  • No meaningful CPU or network load. This is the recommended production setting.

5. What happens if you set tick-ms = 100?

On a machine running only this client, connecting to one server:

Impact at 100 ms
  • 10× more probes per second. Each probe is a round-trip network request, not a local operation.
  • The probe waits up to probe-timeout-ms = 1000 ms for a response. With fixedDelay semantics the scheduler will not re-enter until the previous probe finishes — so if each probe takes 5 ms RTT the effective interval is 100 + 5 = 105 ms, which is fine. But if a probe ever times out (1000 ms) you get back-pressure.
  • Lock contention: monitorConnection() holds the synchronized(this) lock on TcpClientService for the full probe duration. Any simultaneous terminal command (also synchronized) will stall until the probe finishes. At 100 ms ticks this is much more frequent than at 2000 ms.
  • Server load: The server must handle 10 probe messages per second just for keepalive. With only one client this is trivial, but worth knowing.
  • Log volume: client-check.log will receive ~10 entries/sec instead of ~0.5/sec. Over an hour that is ~36 000 lines instead of ~1 800.
  • CPU: On a machine with nothing else running, a Java thread waking every 100 ms is negligible. Scheduler overhead is microseconds. Not a concern here.
  • Reconnection detection improvement: At 2000 ms, if the event-based listener somehow misses a drop, the probe catches it within up to 2 s. At 100 ms the probe would catch it within ≤100 ms. But in practice the event listener detects drops within milliseconds anyway — so the practical gain is near zero.
Verdict

Setting tick-ms = 100 on a single-client, single-server, otherwise-idle machine is safe but wasteful. It adds no meaningful reconnection speed improvement (the event listener already handles that), increases log noise significantly, and raises the chance of transient lock contention during user commands.

Recommended: keep at 2000 ms (or even higher, e.g. 5000) since true drop detection comes from the socket event listener, not the probe tick. The probe is a safety net, not the primary mechanism.
